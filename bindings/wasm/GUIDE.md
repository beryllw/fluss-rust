# fluss-wasm Implementation Guide

## Overview

This document provides a complete guide for implementing and using the Apache Fluss WASM bindings.

## Current Status

⚠️ **Important Limitation**: The current implementation has a dependency on `zstd-sys` (via `opendal`), which requires C compilation and is not compatible with WASM targets.

### Workarounds

1. **For Node.js**: Use the native Node.js addon approach instead of pure WASM
2. **For Production**: Consider using the Java/Scala client via Node.js child processes
3. **Future**: The Fluss team is working on a pure-Rust compression backend

## Architecture

```
fluss-wasm/
├── src/
│   ├── lib.rs          # Main entry point, exports all modules
│   ├── error.rs        # Error types and conversions
│   ├── config.rs       # WasmConfig for client configuration
│   ├── client.rs       # FlussClient main API
│   ├── admin.rs        # FlussAdmin for cluster management
│   └── table.rs        # Table operations (append, scan, upsert, lookup)
├── pkg/                 # Generated by wasm-pack (DO NOT COMMIT)
├── examples/
│   └── usage.js        # Usage examples
├── Cargo.toml          # Rust dependencies
├── package.json        # npm package configuration
└── README.md           # Documentation
```

## Alternative Approaches

### Option 1: Node.js Native Module (Recommended)

Instead of pure WASM, create a Node.js native module using [Neon](https://neon-bindings.com/):

```toml
[dependencies]
fluss = { workspace = true }
neon = "1.0"
tokio = { workspace = true }
```

This approach:
- ✅ Full performance with native code
- ✅ No WASM compilation issues
- ✅ Direct access to Node.js APIs
- ❌ Requires native compilation for each platform

### Option 2: HTTP Proxy Pattern

Create a lightweight HTTP proxy that wraps the Fluss Rust client:

```
┌──────────────┐     HTTP      ┌─────────────┐     TCP     ┌──────────┐
│  Browser/TS  │ ────────────> │ HTTP Proxy  │ ───────────> │  Fluss   │
│   (WASM)     │   JSON/REST   │  (Rust)     │   Protocol  │  Server  │
└──────────────┘               └─────────────┘              └──────────┘
```

**Benefits:**
- ✅ WASM only handles JSON serialization
- ✅ No direct TCP connection needed in browser
- ✅ Single proxy can serve multiple clients

### Option 3: Wait for Pure-Rust Compression

Track these upstream issues:
- [`zstd` pure Rust implementation](https://github.com/gyscos/zstd-rs)
- [`opendal` WASM support](https://github.com/apache/opendal/issues)

### Prerequisites

1. Install Rust nightly toolchain:
```bash
rustup install nightly
rustup default nightly
```

2. Install wasm-pack:
```bash
curl https://rustwasm.github.io/wasm-pack/installer/init.sh -sSf | sh
```

### Build for Node.js

```bash
cd bindings/wasm
npm run build
```

This generates files in `pkg/` directory:
- `fluss_wasm_bg.wasm` - The WebAssembly binary
- `fluss_wasm.js` - JavaScript bindings
- `fluss_wasm.d.ts` - TypeScript definitions

### Build for Web Browsers

```bash
npm run build:web
```

## Key Implementation Details

### 1. Async/Promise Conversion

Rust async functions are converted to JavaScript Promises using `wasm-bindgen-futures`:

```rust
use wasm_bindgen_futures::future_to_promise;

#[wasm_bindgen]
pub fn get_table(&self, database: String, table: String) -> Promise {
    future_to_promise(async move {
        let table_path = TablePath::new(&database, &table);
        let fluss_table = self.inner.get_table(&table_path).await?;
        Ok(JsValue::from(FlussTable::new(fluss_table)))
    })
}
```

### 2. Error Handling

Rust errors are converted to JavaScript exceptions:

```rust
impl From<FlussWasmError> for JsValue {
    fn from(err: FlussWasmError) -> Self {
        JsValue::from_str(&err.to_string())
    }
}
```

### 3. Configuration

Support both string and object configuration:

```javascript
// String (bootstrap servers)
const client = await FlussClient.new("localhost:9123");

// Config object
const config = { bootstrap_servers: "localhost:9123", request_timeout_ms: 30000 };
const client = await FlussClient.new(config);
```

## Known Limitations

1. **No Direct TCP in Browser**: WASM in browsers cannot make raw TCP connections. For browser usage, you need:
   - A WebSocket proxy
   - HTTP gateway
   - Use Node.js environment instead

2. **Tokio Runtime**: Multi-threaded Tokio runtime is not available in WASM. Using single-threaded runtime with `rt` feature.

3. **Filesystem Access**: `storage-fs` feature requires Node.js filesystem access. In browsers, use `storage-memory` or IndexedDB.

## Performance Considerations

1. **Memory Management**: WASM memory is linear and grows automatically. Monitor for memory leaks.

2. **Serialization Overhead**: Converting between JS objects and Rust structures has overhead. Consider batch operations.

3. **Async Boundaries**: Each `future_to_promise` creates overhead. Batch operations when possible.

## Security Considerations

1. **Credentials**: Never hardcode credentials in client-side code
2. **TLS**: Always use secure connections in production
3. **Input Validation**: Validate all JavaScript inputs before passing to Rust
